# AI 投研验证工作台项目说明与实现报告

## 1. 项目概述

本项目是一个面向一级市场投资研究场景的 AI 投研验证工作台。它的核心目标不是简单总结 BP，而是把创业公司 BP、补充材料、财务资料和尽调线索转化为一套可持续更新的结构化投研工作流。

系统围绕投资人实际判断项目质量时关注的问题展开：公司做什么、所在赛道是否成立、业务和技术是否有竞争力、团队和客户信息是否可信、财务质量是否支撑 BP 叙事、当前融资轮次是否合理、已有投资人是否构成有效背书，以及后续还需要哪些材料和访谈来完成验证。

当前版本采用 Python + Streamlit + SQLite 实现，定位为单机或单服务器部署的 MVP。用户可以创建项目、上传 BP、触发 AI 分析、管理验证任务、上传补充材料、查看财务分析和生成综合投研报告。

## 2. 产品解决的问题

传统 BP 初筛依赖投资人手动阅读、做笔记和整理问题，容易出现三个问题：

- BP 信息和真实事实混在一起，难以区分“公司声称”和“已验证事实”。
- 尽调问题分散在聊天记录、表格和文档中，后续补充材料很难自动关联到之前的疑点。
- 项目判断往往停留在主观印象，缺少结构化维度，例如赛道位置、轮次合理性、财务质量和第三方投资人信号。

本项目将 BP 分析拆解成可验证的投研对象，包括 BP 陈述、最重要看点、关键假设、验证任务、直接异常、行业地图、融资轮次判断、财务分析和综合报告。这样用户后续每新增一份材料，系统都可以判断它解决了哪些问题、更新了哪些状态、还留下哪些缺口。

## 3. 核心功能

### 3.1 项目工作台

用户可以在左侧创建、选择和删除项目。每个项目保存公司名称、行业、融资阶段、一句话介绍、当前建议、风险等级、上传材料、结构化分析结果和历史 memo。

主页面采用多页签工作台结构，包括：

- 概览 / BP 上传
- BP 分析
- 行业与产业图
- 轮次与投资人
- 风险与异常
- 验证任务
- 补充材料
- 财务分析
- 综合报告

### 3.2 BP 上传与 AI 初始分析

系统支持上传或粘贴 BP 文本，并支持 TXT、Markdown、PDF、DOCX、PPTX、CSV、XLSX 等文件解析。文档会被切分为带来源页码和 chunk id 的片段，后续所有 AI 输出都必须绑定来源。

BP 初始分析会生成：

- 公司摘要：公司名称、行业、融资阶段、一句话介绍。
- BP 陈述：抽取公司明确表达的事实、判断和数据。
- 最重要看点：提炼 3-7 个值得继续看的积极信号。
- 关键假设：识别项目成立所依赖但尚未充分证明的判断。
- 验证任务：把关键假设转化为可执行的尽调任务。
- 赛道分析：判断一级行业、细分赛道、目标客户和价值链位置。
- 产业图：拆解上游、中游、下游、客户、监管和配套服务，并标注公司所在位置。
- 行业关键词：解释行业术语，降低用户第一次接触新行业时的理解成本。
- 直接异常 / 红旗：单独列出一开始就不符合逻辑或真实性风险明显的问题。
- 融资轮次分析：判断 BP 声称轮次与材料反推轮次是否匹配，分析资料充分度、风险收益、回款周期、估值匹配和已有投资人信号。

### 3.3 行业与产业图

系统会把 BP 中宽泛的行业表述转化为更适合投资判断的细分赛道定义。例如不只停留在“AI 医疗”或“企业服务”，而是进一步判断公司到底处于软件层、数据层、设备层、服务商、集成商还是终端应用环节。

行业与产业图页签展示：

- 一级行业和细分赛道
- 目标客户
- 公司价值链位置
- 替代对象
- 利润池判断
- 产业链节点和公司所在位置
- 行业关键词解释
- AI 术语查询入口

当前版本预留了 `ResearchProvider` 接口，默认不接真实联网搜索；后续可以替换为外部搜索或行业数据库 provider。

### 3.4 融资轮次与投资人分析

系统新增了独立的融资轮次分析模块。它不只是读取项目的融资阶段字段，而是基于 BP 和材料反推当前项目所处阶段，并判断该阶段下材料是否充分。

该模块关注：

- BP 声称轮次和材料反推轮次是否一致。
- 当前资料详细程度是否匹配该轮次。
- 早期项目的高上行和高不确定性。
- 后期项目的稳定性、利润空间、回款周期和财务可验证性。
- 历史投资人、领投方、跟投方、产业方或政府基金是否构成有效背书。
- 历史融资金额、估值、核心条款和资金到账是否需要核验。

这使系统可以把“投资轮次”和“其他投资人信号”纳入项目质量判断，而不是只作为基础信息展示。

### 3.5 风险与异常

系统区分两类风险：

- 信息缺失型风险：例如缺客户合同、缺回款流水、缺访谈，需要进入验证任务。
- 直接异常型风险：例如人员履历冲突、收入口径前后矛盾、客户数量与收入不匹配、资质或融资信息明显可疑，需要作为 red flag 单独展示。

`red_flags` 模块保存异常类型、严重度、状态、来源、影响和建议核验方式。补充材料上传后，如果能解释或反证某个 red flag，系统会更新其状态。

### 3.6 补充材料验证

用户后续上传合同、访谈纪要、财务明细、客户资料、工商资料等补充材料时，系统会把新材料和历史结构化数据进行匹配。

补充材料分析会输出：

- 材料摘要
- 对验证任务的状态更新
- 对 red flag 的解释或反证
- 证据链接
- 补充材料解决了哪些问题
- 如果没有解决历史问题，则总结它新增了什么信息，以及对公司判断有什么价值
- 更新后的阶段性建议和风险等级

这使项目分析从一次性 BP 总结变成持续迭代的尽调过程。

### 3.7 财务分析

系统支持识别财务类材料，包括 CSV、XLSX、财报文本、收入明细、回款记录、成本费用和现金流资料。

财务分析维度包括：

- 收入质量
- 毛利、成本和费用结构
- 现金流和回款质量
- 客户集中度
- 异常波动
- 与 BP 口径的冲突
- 仍需补充的财务材料

财务分析结果会进入财务分析页签，并写入综合投研报告。

### 3.8 综合投研报告

系统可以基于当前所有结构化资料生成 Markdown 综合报告。报告覆盖：

- 项目基本信息
- 赛道归类与产业位置
- 融资轮次与投资人信号
- 产业图摘要
- 最重要看点
- BP 关键陈述
- 直接异常与红旗
- 关键假设与验证任务
- 补充材料解决问题汇总
- 财务分析
- 阶段性建议

报告强调“基于当前证据充分度”的阶段性判断，不把 AI 输出包装成最终投资决策。

## 4. 技术架构

### 4.1 前端与交互层

前端使用 Streamlit 实现，入口文件是 `app.py`。它负责：

- 页面布局和项目切换
- 文件上传和文本粘贴
- 展示结构化分析结果
- 调用 BP 分析和补充材料验证
- 提供 Markdown 报告预览和下载

Streamlit 的优点是开发速度快、适合本地或单服务器 MVP，也便于在投研工具早期快速迭代业务流程。

### 4.2 文档解析层

文档解析逻辑位于 `services/parser.py`。系统根据文件类型调用不同解析方式：

- TXT / Markdown / JSON：文本读取
- PDF：`pypdf`
- DOCX：`python-docx`
- PPTX：`python-pptx`
- CSV：标准库 `csv`
- XLSX：`openpyxl`

解析后的文本会通过 `make_chunks` 切分为 chunk。每个 chunk 保存：

- chunk id
- chunk index
- page number
- section label
- text

section label 会根据关键词粗分为团队、客户与需求、经营数据、市场、融资、产品与技术或正文，帮助后续 AI prompt 理解上下文。

### 4.3 AI 分析层

AI 分析逻辑位于 `services/pipeline.py`，主要入口包括：

- `analyze_bp_batch`：BP 初始分析或追加 BP 后重新综合分析。
- `verify_supplementary`：补充材料验证。
- `build_report_from_bundle`：基于当前结构化数据生成综合报告。
- `explain_industry_term`：根据项目上下文解释行业术语。

LLM 调用由 `services/llm.py` 封装，支持 OpenAI-compatible API，通过环境变量配置：

- `LLM_BASE_URL`
- `LLM_API_KEY`
- `LLM_MODEL`

AI 输出采用严格 JSON schema，并通过校验函数检查字段完整性、枚举值和来源 chunk 是否有效。这样可以降低模型自由发挥导致的数据污染。

### 4.4 数据持久化层

数据存储使用 SQLite，逻辑集中在 `services/storage.py`。当前主要表包括：

- `projects`：项目基本信息。
- `documents`：上传文档记录。
- `document_chunks`：解析后的文档片段。
- `bp_claims`：BP 陈述。
- `key_highlights`：最重要看点。
- `investment_assumptions`：关键假设。
- `verification_tasks`：验证任务。
- `evidence_links`：补充材料与任务/陈述之间的证据链接。
- `sector_analyses`：赛道分析。
- `industry_map_nodes`：产业图节点。
- `industry_terms`：行业关键词。
- `red_flags`：直接异常和红旗。
- `funding_analyses`：融资轮次与投资人分析。
- `supplementary_resolutions`：补充材料解决问题汇总。
- `financial_analyses`：财务分析。
- `investment_memos`：历史 memo / 报告内容。

所有项目相关数据都通过 `project_id` 关联，删除项目时会级联删除相关数据。

## 5. 关键实现机制

### 5.1 来源绑定

系统要求 BP 陈述、最重要看点和 red flags 必须绑定 `source_chunk_id` 和 `source_page`。这使用户可以追溯每个判断来自哪一页、哪一段材料，避免 AI 生成无法追溯的结论。

### 5.2 状态驱动的验证工作流

验证任务和 BP 陈述都有状态，包括：

- 未验证
- 部分验证
- 已验证
- 无法验证
- 被反证

补充材料上传后，系统会根据 AI 判断更新任务、陈述和假设状态，并写入证据链接。red flag 也有独立状态，包括未解决、部分解决、已解决和被反证。

### 5.3 初始分析与补充材料分工

系统把 BP 初始分析和补充材料验证分成两个不同工作流：

- 初始分析负责建立项目的结构化底稿。
- 补充材料验证负责更新状态、补充证据和解释既有疑点。

这种设计更符合真实投研流程：先形成问题清单，再用材料逐步验证。

### 5.4 保留用户手动判断

用户可以手动添加验证任务、修改任务状态和备注。BP 重新分析时，系统会替换 AI 自动生成的结构化分析，但保留用户手动创建的任务，避免覆盖人工尽调判断。

### 5.5 可扩展的行业研究接口

`services/research_provider.py` 定义了 `ResearchProvider` 协议。当前版本使用 `NullResearchProvider`，表示第一版不接真实外部搜索，只保留接口形态。未来可以接入：

- 搜索引擎 API
- 行业数据库
- 政策/监管数据源
- 融资数据库
- 企业工商和知识产权数据源

这样后续增强行业研究时，不需要重写主流程。

## 6. 当前项目能力边界

当前版本已经具备完整 MVP 闭环，但仍有明确边界：

- 没有登录注册和权限体系，适合本地或单用户部署。
- 行业研究暂未接真实联网搜索，主要依赖 BP 内容和 LLM 归类。
- 图片 OCR 尚未接入。
- 输出报告是 Markdown，不是正式 PDF 或 Word 版式文件。
- LLM 输出虽然有 schema 校验，但仍需要用户进行专业复核。

## 7. 后续可扩展方向

后续可以继续增强以下方向：

- 接入外部行业搜索和来源引用，形成有来源链接的行业研究报告。
- 接入工商、诉讼、知识产权、融资数据库，增强真实性核验。
- 将资料清单转化为动态尽调清单，按轮次和行业自动推荐应补材料。
- 增加 Word / PDF 报告导出。
- 增加多项目比较、项目评分和投委会 memo 模板。
- 增加用户、权限和团队协作能力。
- 增加文件级证据定位和原文高亮。

## 8. 总结

本项目实现的是一个以 BP 为起点、以验证任务为核心、以补充材料持续更新为机制的 AI 投研工作台。它将投资人对项目质量的判断维度结构化，包括主体治理、团队、业务技术、行业位置、融资轮次、投资人信号、财务质量、风险异常和证据充分度。

从实现上看，系统采用 Streamlit 快速构建交互界面，以 SQLite 保存项目级结构化数据，通过文档解析和 chunk 来源绑定保证材料可追溯，再用 OpenAI-compatible LLM 将非结构化 BP 和补充材料转化为可验证、可更新、可生成报告的投研资产。

该项目的价值不在于替代投资人做最终决策，而在于把“看 BP 的主观判断”转化为“可追溯、可验证、可迭代的投研工作流”。
